# 内容获客：文章模块与分类功能 — 扩展实现说明

本文说明本项目中 **文章（ContentArticle）** 与 **文章分类（ContentArticleCategory）** 是如何接入的，以及 **左侧菜单**、**数据库表** 是如何出现的。可作为下次增加类似业务模块时的操作手册。

---

## 1. 整体思路（你要记住的三条线）

| 线 | 作用 |
|----|------|
| **① 数据库** | 定义 `model`，在业务库 **AutoMigrate** 里注册 → **启动后端** 自动建表/加列。 |
| **② 后端 HTTP** | `router` 注册路由 → `api` 处理请求 → `service` 写业务 → 操作 `model` 对应表。 |
| **③ 菜单与权限** | 通过 **初始化/同步脚本** 写入 `sys_base_menus`、`sys_apis`、Casbin 规则；管理端点一次 **「同步内容获客」** 增量写入，**重新登录** 后左侧才稳定出现菜单与接口权限。 |

前端管理页：在 `web/src/view/...` 写 Vue，在 `web/src/api/...` 封装请求；菜单路径与 **后端返回的动态路由** 或 **同步进库的 component 路径** 一致即可。

---

## 2. 数据库表是怎么「加出来」的

### 2.1 模型定义（Model）

- 文章：`server/model/content/content_article.go` → 表名 `content_articles`（`TableName()`）。
- 分类：`server/model/content/content_article_category.go` → 表名 `content_article_categories`。
- 文章上增加外键字段：`category_id`（在 `ContentArticle` 结构体里），并可选 `Preload` 关联 `Category`。

### 2.2 注册到 AutoMigrate（业务库）

业务表不在 `initialize/gorm.go` 的主 `RegisterTables()` 大列表里逐条写死，而是放在 **业务初始化** 里：

- 文件：`server/initialize/gorm_biz.go`
- 函数：`bizModel()` → `db.AutoMigrate(&content.ContentArticle{}, &content.ContentArticleCategory{}, ...)`

主入口 `server/initialize/gorm.go` 在 `RegisterTables()` 末尾会调用 **`bizModel()`**，因此只要：

1. 配置里 **没有** 关闭自动迁移（`disable-auto-migrate` 不为 `true`）；
2. **正常启动** `server`（如 `go run .`）；

就会在**当前连接的业务库**里自动创建/更新表结构（含新增列）。

### 2.3 若关闭了 AutoMigrate

需自行执行 SQL 建表，或临时打开自动迁移启动一次再关。

## 3. 后端接口是怎么挂上的

### 3.1 路由入口（谁把路由挂到 Gin 上）

- 总路由：`server/initialize/router.go` → `initBizRouter(PrivateGroup, PublicGroup)`。
- 业务路由注册：`server/initialize/router_biz.go` → 调用 `content` 包下的 `InitContentArticleRouter`。

### 3.2 文章 + 分类（私有 / 公开）

- 文件：`server/router/content/content_article.go`
- **私有组**（需 JWT + Casbin）：如 `contentArticle/*`、`contentArticleCategory/*`。
- **公开组**（无登录）：`public/*` 下的文章列表、按 slug 查文章等（与分类管理无关）。

### 3.3 分层对应关系（复制新业务时的模板）

| 层级 | 目录/说明 |
|------|-----------|
| Model | `server/model/content/*.go` |
| Service | `server/service/content/*.go`，在 `enter.go` 的 `ServiceGroup` 里挂上 `XxxService` |
| API | `server/api/v1/content/*.go`，在 `api/v1/content/enter.go` 的 `ApiGroup` 里挂上 `XxxApi`，并 `var xxxService = service.ServiceGroupApp....` |
| Router | `server/router/content/*.go`，在 `router/content/enter.go` 里 `var xxxApi = api.ApiGroupApp...`，在 `InitXxxRouter` 里 `private.Group(...)` |

**注意**：同包 `service` 内若 `ArticleService` 要调用 `ArticleCategoryService`，可用 `service/content/enter.go` 里的包级变量（本项目中 `articleCategoryService`），避免与 `api` 包里的命名混用即可。

## 4. 左侧菜单是怎么出现的

Gin-Vue-Admin 左侧菜单来自 **数据库** `sys_base_menus`（及角色-菜单关联），不是纯前端写死。

本项目中「内容获客 → 文章管理」一类菜单，是通过 **同步脚本** 写入的：

- 脚本：`server/source/content/sync.go` → 函数 `SyncContentInit(db *gorm.DB)`  
  - 写入/补全 **父菜单**（如 `content`）、**子菜单**（如 `contentArticle`，`component` 指向 `view/content/article/index.vue`）。  
  - 写入 **SysApi**（各 Method + Path）。  
  - 写入 **Casbin** `p` 规则（角色 `888` / `8881` / `9528` 等与路径、Method 对应）。

**触发同步的入口（管理端 UI）：**

- 页面：`web/src/view/systemTools/system/system.vue`（系统配置页底部按钮）。  
- 按钮文案：**「同步内容获客菜单/权限」**。  
- 调用：`web/src/api/contentInit.js` → `POST /contentInit/sync`。  
- 后端：`server/api/v1/system/content_init_sync.go` → `ContentInitService.Sync()` → 内部执行 `contentSource.SyncContentInit(global.GVA_DB)`。

**操作顺序建议：**

1. 先保证后端已启动过，**表已 AutoMigrate**（见第 2 节）。  
2. 超级管理员登录管理端 → **系统工具 → 系统配置**。  
3. 点击 **「同步内容获客菜单/权限」**，确认执行。  
4. **退出并重新登录**（或按提示刷新权限），左侧即可看到「内容获客」下的「文章管理」等。  
5. **分类**：当前实现是挂在 **文章管理页** 内的 **「分类管理」** 按钮，**没有**再单独加一条左侧「分类」菜单；若需要独立菜单，要在 `sync.go` 里再 `Create` 一条 `SysBaseMenu`（`component` 指向新页面），并再执行一次同步。

**新接口要能在角色里勾选：** 必须把 Path + Method 写进 `sync.go` 的 `apis` 与 `paths`（Casbin）两段循环里，否则同步不会插入 API / 权限，接口会 403。

## 5. 前端管理端（文章 + 分类）落点

| 功能 | 说明 |
|------|------|
| 文章列表/编辑 | `web/src/view/content/article/index.vue` |
| 文章 API | `web/src/api/contentArticle.js` |
| 分类 API | `web/src/api/contentArticleCategory.js` |
| 动态路由 | 菜单里 `component` 为 `view/content/article/index.vue` 时，由框架异步加载对应页面（与 `pathInfo.json` 等机制配合，按项目版本为准）。 |

**分类与文章的关联：** 表单里 `categoryId` 与级联选择器绑定；保存时随 `createContentArticle` / `updateContentArticle` 提交；列表展示 `Preload("Category")` 后的 `category.name`。

---

## 6. 分类业务规则摘要（便于对照实现）

- 表：`content_article_categories`，`parent_id = 0` 为一级；子级 `parent_id` 指向一级 ID → **最多二级**。  
- 服务层禁止在「二级」下再建子节点；删除前检查子分类与 `content_articles.category_id` 引用。  
- `sync.go` 在分类表为空时可插入示例数据（可选，便于第一次打开就有下拉项）。

## 7. 下次加「另一个业务模块」时的推荐清单

可按顺序打勾：

1. [ ] **Model**：`server/model/<包>/<实体>.go`，写好 `TableName()`、字段、`json` 标签。  
2. [ ] **AutoMigrate**：在 `server/initialize/gorm_biz.go`（或项目约定的业务 migrate 入口）注册模型。  
3. [ ] **Service**：`server/service/<包>/`，并在 `enter.go` 的 `ServiceGroup` 注册。  
4. [ ] **API**：`server/api/v1/<包>/`，并在 `api/v1/<包>/enter.go` 绑定 `ServiceGroupApp`。  
5. [ ] **Router**：`server/router/<包>/`，在 `initialize/router_biz.go`（或同类文件）里 `Init...Router`。  
6. [ ] **同步脚本**（若需要菜单/权限/API）：在 `server/source/<业务>/sync.go` 或扩展现有 `SyncContentInit`：  
   - 菜单 `SysBaseMenu`；  
   - `SysApi` 列表；  
   - Casbin `paths` 与角色循环。  
7. [ ] **管理端页面 + API**：`web/src/view/...`、`web/src/api/...`。  
8. [ ] **执行一次同步**（或菜单管理里手工加菜单 + API 管理里手工加 API + 角色授权）。  
9. [ ] **重启后端、重新登录**，验证列表、表单、权限。
